# v0.7.2 Unified Conversation Report Plan

## Goal

Each conversation must produce exactly one report file:

```text
reports/oa-report-<timestamp>-<session>.json
```

That one report must be understandable and useful for:

1. Doctors and medical researchers.
2. Engineers debugging the system.
3. AI evaluation and iterative improvement pipelines.

The report should not split into separate dossier, training, or debug files. If any party cannot understand or use the single report, the report schema is wrong.

## Unified Report Shape

The top-level report should preserve existing clinical fields for compatibility, then add organized sections for review and AI iteration:

```json
{
  "report_type": "oa_home_pain_check_in",
  "schema_version": "0.7.2",
  "assistant_version": "v0.7.2",
  "generated_at": "...",

  "session": {},
  "patient_identity": {},
  "readiness": {},
  "pain_assessment": {},
  "safety_assessment": {},
  "doctor_summary": "...",
  "suggested_follow_up_priority": "...",
  "limitations": [],

  "medical_research_review": {},
  "conversation_trace": {},
  "ai_iteration": {},
  "audit_metadata": {},

  "transcript": []
}
```

This section should explain the relay of information without forcing reviewers to inspect logs:

```json
{

  "turns": [

      "role": "assistant",

    {
      "turn_index": 2,
```

## AI Iteration

This section is not automatically approved training data. It is a queue of reviewable candidates:

```json
{
  "review_status": "unreviewed",
  "approved_for_training": false,
  "failure_tags": [
    "reply_too_static",
    "rulebook_overclarified"
  ],
  "training_candidates": [
    {
      "type": "response_preference",
      "requires_human_review": true,
      "reason": "Assistant repeated a static prompt after patient confusion.",
      "context": [],
      "preferred_behavior": "Acknowledge confusion, then ask the same clinical item in simpler wording."
    }
  ]
}
```

## Implementation Plan

### Step 1: Extend Existing Report Only

- Update `app/report.py`.
- Do not add `app/dossier.py`.
- Do not add `/api/save_dossier`.
- Keep `/api/save_report` as the single report-saving endpoint.

### Step 2: Add Structured Sections

- `medical_research_review`
- `conversation_trace`
- `ai_iteration`
- `audit_metadata`

### Step 3: Derive Metrics From Existing State

- Use `state.transcript` for turn counts and clarification hints.
- Use `state.model_events` for STT/model/TTS visibility.
- Use clinical state for completeness and research quality.

### Step 4: Add Tests

- Completed conversation report contains all new sections.
- Only one report artifact is generated by `/api/save_report`.
- Report includes model trace and training candidates.
- Raw audio is not included by default.

## Non-Goals

- No automatic RL training.
- No raw audio retention by default.
- No second report file per conversation.
- No model authority over clinical flow.

## Success Criteria

One saved report is enough for:

1. A doctor to understand the patient status.
2. A medical researcher to assess data quality.
3. An engineer to debug STT/LLM/TTS behavior.
4. An AI pipeline to extract human-reviewable improvement examples.
