# v0.6

## Summary

Adds an experimental hybrid Covo half-duplex voice mode while keeping the deterministic OA clinical engine authoritative.

## Goal

v0.6 tests whether a half-duplex voice-to-voice style workflow can improve UX without giving up clinical correctness:

- record one patient spoken turn at a time
- optionally send audio to a Covo half-duplex service
- pass the resulting patient transcript into the existing OA state machine
- speak the next required clinical prompt using the existing speech fallback path
- preserve deterministic red-flag escalation and doctor report generation

## Architecture

```text
browser recorded patient turn
  -> /api/covo/turn
  -> optional external Covo service
  -> patient transcript
  -> ConversationEngine.handle_user_message(...)
  -> required next OA prompt
  -> server/browser speech output
```

Covo is treated as a voice interface. The OA app remains the clinical controller.

## Backend Changes

- Added `app/covo.py` with a lightweight HTTP client for a future Covo service.
- Added Covo status metadata under `/api/voice_status`.
- Added `/api/covo/start` to create a deterministic OA clinical session for the Covo experiment.
- Added `/api/covo/turn` to accept a recorded audio turn and optional browser transcript.
- `/api/covo/turn` only calls Covo when `ENABLE_COVO=1` and `COVO_ENDPOINT` is configured.
- Without a configured Covo service, the endpoint can still exercise the same half-duplex OA flow using browser transcript capture.

## Frontend Changes

- The primary start button now launches the Covo half-duplex experiment.
- The second call button toggles recording a single patient answer.
- Browser `MediaRecorder` captures the audio turn.
- Browser speech recognition captures a fallback transcript for local testing while GPUs are unavailable.
- Existing typed fallback remains available.

## Configuration

```bash
export ENABLE_COVO=1
export COVO_ENDPOINT="http://api.datummed.com:PORT"
export COVO_MODEL_DIR="/hdd-storage/lawrencelcty/huggingface/models/Covo-Audio/covoaudio"
export COVO_DEVICE="cuda:2,cuda:3"
export COVO_TIMEOUT_SECONDS=120
```

Expected Covo service endpoint:

```text
POST {COVO_ENDPOINT}/turn
```

The request contains base64 patient audio, language, session id, the last required clinical prompt, and current OA state. The response should contain a patient transcript and may optionally contain generated audio.

## Known Limitations

- Tencent's public Covo-Audio release does not include `Covo-Audio-Chat-FD` runtime assets or full-duplex server code.
- This v0.6 implementation is half-duplex and turn-based.
- The main app does not load the 7B Covo model directly.
- GPU validation is pending because GPUs 2 and 3 are currently occupied by another user's VLLM processes.
- Until a Covo service is running, the Covo UI path uses browser transcript capture to test state-machine integration.
