# v0.5

## Summary

Adds an OpenAI Realtime-first voice architecture and a local Qwen3-TTS fallback, then records the UX finding that neither path is sufficient for stakeholder-facing full voice interaction. The next direction is a true voice-to-voice realtime model.

## Goal

v0.5 attempted to improve the voice-call experience beyond the browser speech and hybrid v0.4 flow:

- Use OpenAI Realtime over WebRTC as the primary low-latency speech-to-speech path.
- Keep the deterministic clinical engine in control of protocol progression, safety escalation, and report generation.
- Add local Qwen3-TTS speech synthesis as a fallback when OpenAI is unavailable.
- Keep the rule-based scripted/browser flow only as the final fallback.

## User-Facing Changes

- The first-choice call path is OpenAI Realtime with live microphone streaming and streamed assistant audio.
- If OpenAI Realtime cannot start, the app falls back to the local deterministic check-in flow.
- In fallback mode, the app now tries backend `/api/tts` audio before browser speech synthesis.
- If backend local speech synthesis fails, the app falls back to browser speech/typed input.
- Status copy now distinguishes realtime voice, local Qwen voice fallback, and base browser fallback.

## Backend Changes

- Added Qwen3-TTS fallback support in `app/tts.py`.
- Default Qwen3-TTS model path:


- Added lazy Qwen3-TTS loading so the app can start without loading the model immediately.

- Added configurable Qwen TTS settings:
  - `ENABLE_QWEN_TTS`

- Kept OpenAI TTS as the first `/api/tts` attempt when OpenAI TTS is explicitly enabled.
- Kept Kokoro only as an optional legacy fallback.

## Frontend Changes

- Added server-audio fallback playback in `static/app.js`.
- Realtime startup failure now routes to local server voice mode first.
- `speakFallback()` now tries:
  1. `/api/tts`
  2. browser `speechSynthesis`
  3. typed/text-only interaction if speech is unavailable
- Added fallback audio lifecycle cleanup when ending/restarting a call.

## Safety / Clinical Rule Changes

- The voice layer only changes how prompts are spoken, not what clinical content is required.

## Known Limitations

- OpenAI Realtime did not provide sufficient UX for stakeholders.
- Qwen3-TTS is text-to-speech only, not a voice-to-voice dialogue model.
- The Qwen fallback still uses the pipeline:

```text
mic -> STT / text answer -> deterministic clinical engine -> assistant text -> TTS audio
```

- This pipeline still has turn-taking latency and does not provide true full-duplex voice interaction.
- Actual Qwen synthesis could not be verified in the sandbox because the Python process cannot see an NVIDIA driver.
- The host still needs the system `sox` binary if Qwen runtime paths require it.
- `flash-attn` is not installed, so Qwen inference may be slower and use more memory.
- Tracked `__pycache__` files changed during local compile checks and should be reviewed before commit.

## Decision / Pivot

The stakeholder-facing UX target now requires a true V2V architecture rather than STT + LLM/rules + TTS.

The next prototype should evaluate a full-duplex audio-in/audio-out model similar to Tencent CovoAudio-style systems. The clinical state machine should remain the source of truth, but the speech loop should move to a persistent realtime audio session with:

- continuous microphone streaming
- streamed assistant audio output
- barge-in/interruption handling
- side-channel tool calls into `submit_patient_answer`
- deterministic clinical state/report updates from this app

## Follow-Up Candidates

- Add a new V2V adapter boundary separate from the current `/api/tts` fallback.
- Define a realtime tool contract compatible with both OpenAI Realtime and future V2V servers.
- Decide whether browser audio should connect directly to the GPU voice server or proxy through this app.
- Prototype LAN GPU-server latency before committing to a cloud/server deployment.
- Keep Qwen3-TTS as a non-primary local TTS fallback only.
