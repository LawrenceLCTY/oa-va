# v0.4.2

## Summary

Adds guided VAS/NRS pain-score calibration using binary comparisons against concrete osteoarthritis pain scenarios.

## Goal

Directly asking a participant to provide a 0 to 10 pain score can be unreliable, especially in voice calls and older-adult monitoring. v0.4.2 keeps the final 0 to 10 score, but derives it through anchored comparisons when possible:

- Present two pain-level descriptions.
- Ask whether the participant's pain is closer to the milder or stronger description.
- Narrow the range using binary filtering.
- Arrive at a calibrated score within about 3 to 4 answers.

## User-Facing Changes

- Average 24-hour pain and current pain now start with an anchored comparison flow.
- The app describes pain using concrete OA-relevant scenarios such as mild discomfort, slowed stairs/walking/hand use, difficulty with daily tasks, and extreme pain.
- Participants can still provide a direct number from 0 to 10; the app accepts it as a direct numeric override for backward compatibility and for confident respondents.
- Clarification prompts now ask whether pain is closer to the first milder description or the second stronger description.
- If OpenAI Realtime or another AI voice path is unavailable, the browser falls back to the deterministic scripted conversation path with typed input and browser speech where available.

## Backend Changes

- Added `app/vas.py` with:
  - 0 to 10 pain anchors in English and Chinese
  - binary comparison question generation
  - lower/higher answer parsing
  - range narrowing
  - final score summary
  - trace-entry generation
- Extended `PainAssessment` in `app/schemas.py` with active VAS state and calibration traces.
- Updated `ConversationEngine` to use VAS calibration for:
  - `average_pain_score`
  - `current_pain_score`
- Added report fields for:
  - `average_24h_vas_trace`
  - `current_vas_trace`
- Updated report scale label to `0-10 VAS/NRS with optional binary scenario anchoring`.
- Added frontend fallback mode that starts `/api/start` and sends replies through `/api/message` when `/api/realtime/start` fails.
- Added `/api/voice_status` fallback capability metadata.

## Safety / Clinical Rule Changes

- No red-flag rule changes.
- No emergency escalation changes.
- Severe pain priority logic still uses the final numeric score.
- Direct numeric answers remain valid, but binary calibrated scores include trace data for auditability.

## Known Limitations

- The pain anchors are initial clinical/UX heuristics and should be reviewed by a clinician.
- Binary answer parsing is intentionally simple and may need expansion after real voice transcripts are observed.
- The flow can still accept direct numeric scores, so calibrated scoring is encouraged but not mandatory.
- Browser speech recognition and speech synthesis support varies by browser; typed input remains the guaranteed fallback.

## Follow-Up Candidates

- Clinician review of all 0 to 10 anchor descriptions.
- Add transcript examples from real calls to improve lower/higher parsing.
- Add UI visualization of the narrowed pain range for text-assisted sessions.
- Consider storing whether each score was `binary_anchor_calibrated` or `direct_numeric_override` as a structured field rather than only in `patient_words`.
