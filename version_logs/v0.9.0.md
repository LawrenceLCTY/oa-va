# v0.9.0

Date: 2026-07-22

## Goal

Replace the short OA pain check-in as the active flow with a DOCX/audio-guided osteoarthritis medication and treatment questionnaire.

## Source Materials

- `data/OA问卷-房山-简化-v2.docx`
- `data/新录音 4.m4a`
- `data/新录音 4.txt`

The DOCX defines the required fields. The audio transcript defines the target phone-interview style: natural wording, one answer per turn, field-by-field confirmation, and conditional branch skipping.

## User-Facing Changes

- Starts as an OA medication/treatment questionnaire instead of a short pain check-in.
- Collects all DOCX questionnaire fields through one-question-at-a-time voice prompts.
- Uses conditional branching for oral analgesic follow-ups, missed-dose reasons, adverse-reaction details, hospital/community counseling, and retail-pharmacy questions.
- Updates UI labels and report preview around questionnaire completion and structured survey responses.

## Backend Changes

- Added a deterministic questionnaire definition/parser layer in `app/questionnaire.py`.
- Added questionnaire storage to `ConversationState`, including normalized answers, raw answers, skipped conditional fields, completion metadata, and source-material metadata.
- Wired questionnaire steps into `ConversationEngine` as the default v0.9.0 path.
- Updated report output to `report_type=oa_medication_treatment_questionnaire`, `schema_version=0.9.0`, and `assistant_version=v0.9.0`.

## Safety/Clinical-Rule Changes

- Existing red-flag detection remains active during all turns.
- A final red-flag screen remains after the questionnaire.
- The assistant still does not diagnose, prescribe, or adjust medication.

## Validation Changes

- Added deterministic parsing for yes/no, 0-10 flare pain score, flare-frequency buckets, single-choice handling behavior, multi-choice affected joints/treatments/channels/reasons, improvement ratings, willingness, doctor counseling, pharmacy purchase method, and pharmacy guidance frequency.
- Stores raw participant words alongside normalized values for review.

## Known Limitations

- The parser is deterministic and keyword-based; unusual answers may still require clarification.
- The source files in `data/` are ignored by git and must be present locally for review context.
- The implementation follows the questionnaire content and sample-interview style but is not a validated survey instrument.
