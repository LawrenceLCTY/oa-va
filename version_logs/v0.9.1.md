# v0.9.1

Date: 2026-07-22

## Goal

Improve the private local voice UX by reducing server TTS wait failures and removing the worst manual turn-taking friction after the v0.9.0 DOCX questionnaire merge.

## User-Facing Changes

- Added click-to-interrupt behavior for private voice mode: starting a patient turn stops any active assistant speech first.
- Added automatic private-mode listening after assistant speech finishes.
- Added browser-side endpointing so a recorded answer can auto-stop after detected speech followed by silence.
- Combined same-turn assistant messages into one spoken TTS request to reduce choppy multi-utterance playback.
- Replaced the misleading local-TTS failure message with a clearer temporary server-voice fallback message.
- Server TTS now falls back to browser speech per response instead of permanently downgrading after one transient miss.

## Backend Changes

- Added a warmed TTS sidecar in `app/tts_server.py` with `/api/health`, `/api/tts`, and `/api/warmup`.
- Updated `app/tts.py` to route through `TTS_SERVICE_URL` when configured, with in-process fallback.
- Added TTS latency and engine headers from the main `/api/tts` adapter.
- Loaded `.env` in `app/qwen_server.py` so local model settings are available when starting the extractor server.
- Warmed the actual opening questionnaire prompts on TTS sidecar startup when `TTS_WARMUP_ON_START=1`.
- Increased `TTS_SERVICE_TIMEOUT_SECONDS` from 3 seconds to 20 seconds to handle first uncached CosyVoice prompts.

## Environment/Dependency Notes

- CosyVoice was kept in the isolated `tts` conda environment rather than installing its full stack into `oava`.
- The working TTS environment used Torch/Torchaudio CUDA 12.1 builds and NumPy 1.26.x to avoid the ONNX Runtime NumPy 2.x ABI failure.
- CosyVoice3 required the `inference_instruct2` instruction end marker and the local CosyVoice `torchaudio.load` backend adjustment for the available runtime.

## Safety/Clinical-Rule Changes

- None. v0.9.1 changed the voice transport and turn-taking UX, not the questionnaire protocol or clinical rule authority.

## Known Limitations

- This is still not true always-on full-duplex barge-in. It provides immediate click-to-interrupt and automatic half-duplex turn continuation.
- First uncached CosyVoice prompts may still take several seconds; warmup and caching reduce demo-visible latency but do not make CosyVoice streaming.
- Browser VAD thresholds are heuristic and may need tuning in noisy rooms or with laptop speakers.

## Follow-Up Candidates

- Add always-on VAD with echo-safe barge-in detection.
- Add explicit model-health gating before starting a demo session.
- Add per-turn latency metrics in the report for STT, clinical engine, TTS, and playback wait time.
