# v0.7.2 Development Plan

## Theme

Conversation reporting for medical review and AI iteration.

## Motivation

v0.7.1 can run the private model stack, but the assistant can still feel static. The next step is not simply adding another model. The system needs better evidence from each conversation so the team can identify exactly what failed and turn that into reviewable training/evaluation data.

## Planned Deliverables

- Upgrade the existing conversation report into one unified JSON artifact.
- Keep exactly one saved report file per conversation.
- Add turn-level records for:
  - assistant required prompt
  - assistant spoken prompt
  - patient transcript
  - STT source and timing
  - local Qwen extraction trace
  - rulebook decision
  - TTS engine and fallback status
  - reviewer failure tags
- Add medical-research quality metrics:
  - required field completeness
  - suspect or inconsistent answers
  - high-pain and red-flag review triggers
  - study usability status
- Add AI-iteration artifacts:
  - extractor correction candidates
  - response preference candidates
  - failure tags for STT, extraction, rulebook, rewrite, TTS, and latency
- Add review states:
  - `unreviewed`
  - `medical_reviewed`
  - `ai_reviewed`
  - `approved_for_training`
  - `excluded_from_training`

## Planned Files

- `docs/v0.7.2-conversation-reporting-plan.md`
- updates to `app/report.py`
- new tests for unified report sections and trace completeness

## Non-Goals

- No automatic RL training in v0.7.2.
- No raw audio retention by default.
- No model authority over clinical flow.
- No separate dossier/training files per conversation.

## Success Criteria

Each completed conversation should produce one JSON report containing:

1. A doctor-readable report.
2. A medical-research review summary.
3. A turn-by-turn AI trace.
4. Searchable failure tags.
5. Human-reviewable training candidates.
