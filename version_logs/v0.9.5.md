# v0.9.5

Date: 2026-07-23

Status: Implemented baseline. Further tuning can be appended after live-call testing.

## Goal

Make the local AI the first semantic interpreter after STT. The deterministic questionnaire code should validate and store structured results, not try to understand culturally fuzzy human speech by keyword rules.

## Correct Turn Stack

```text
Browser audio / fallback text
-> STT transcript sanitation
-> Qwen semantic turn interpreter
-> deterministic validator / safety gate
-> state transition
-> required next prompt
-> TTS / browser speech
```

## Qwen Interpreter Contract

Create a dedicated questionnaire-turn interpreter, separate from generic safety extraction. It must receive:

- current step key
- exact current question text
- expected answer type
- allowed options / schema for the current step
- conditional branch context from already-collected answers
- recent transcript
- current human response

It must return JSON only:

```json
{
  "turn_type": "answer | correction | complaint | off_topic | safety | unclear",
  "answers_current_question": true,
  "normalized": {"value": null, "values": [], "other_text": null},
  "confidence": 0.0,
  "evidence": "",
  "reason": "",
  "clarification_prompt": null,
  "safety_flags": []
}
```

## Rulebook Responsibility

The rulebook becomes a validator, not an interpreter. It should only:

- check the Qwen output matches the expected schema for the step
- enforce allowed option values
- enforce required/conditional branches
- preserve safety escalation rules
- reject low-confidence or malformed model output
- store raw evidence plus normalized output

The rulebook should not keep growing fuzzy keyword interpretations except for hard safety checks and simple deterministic sanity checks.

## Acceptance Policy

Advance the questionnaire only when all are true:

- `turn_type == "answer"`
- `answers_current_question == true`
- normalized value passes validator
- confidence is above threshold, initially `>= 0.72` for ordinary fields and higher for safety-critical fields

Clarify without advancing when:

- complaint/correction/off-topic
- model says unclear
- confidence below threshold
- normalized output fails schema validation
- model unavailable and deterministic fallback cannot confidently parse the answer

## Safety Policy

Safety remains outside normal questionnaire flexibility:

- red-flag keyword/rule detection still runs before and after Qwen
- Qwen may add safety flags but cannot suppress rule-detected red flags
- urgent escalation bypasses questionnaire advancement
- reports must include model confidence/evidence for every AI-interpreted answer

## Current Limitations

- Semantic quality now depends on the local Qwen server actually running with acceptable latency. The v0.9.5 environment check passed required configuration checks; Qwen/TTS sidecar health checks were optional and not running during validation.
- The fallback parser remains for offline/dev resilience; production tests should run with `ENABLE_LOCAL_UNDERSTANDING=1` and a healthy Qwen endpoint.
- The validator currently uses one global confidence threshold for questionnaire answers; safety-critical or high-risk fields may need per-step thresholds after testing.
