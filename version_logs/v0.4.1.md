# v0.4.1

## Summary

Voice-first UX refinement focused on making the scripted clinical questions warmer while keeping the deterministic protocol clear and enforceable.

## Goal

The v0.4 script was clinically direct, but several questions sounded mechanical in spoken conversation. v0.4.1 separates the clinical purpose of each question from the patient-facing wording, so the app can ask in a more humane way without weakening validation or report structure.

## User-Facing Changes

- Rewrote the English and Chinese protocol questions to sound more natural in a live voice call.
- Kept questions short, direct, and slot-specific so patients still know exactly what information is needed.
- Improved bilingual wording for identity confirmation, respondent source, pain scoring, pain location, functional impact, treatment context, side effects, medication changes, emergency visits, and red-flag screening.
- Preserved the Beijing University research-team reference in the Chinese opening.
- Kept text as a supplemental transcript/backup path while prioritizing voice interaction.

## Backend Changes

- Added `PROMPT_INTENTS` in `app/i18n.py` to record the stable clinical intent behind each prompt.
- Added `intent()` accessor in `app/i18n.py`.
- Updated Realtime spoken instructions in `app/main.py` so model-facing instructions include both:
  - the friendly required spoken line
  - the underlying clinical intent for the current protocol step
- Kept the existing conversation state machine and validators as the enforcement layer.

## Safety / Clinical Rule Changes

- No clinical rule changes.
- No validator logic changes.
- No report schema changes.
- Red-flag detection and escalation remain deterministic.
- The wording is friendlier, but the required clinical meaning, score ranges, and safety wording are preserved.

## Known Limitations

- The prompt intent map is currently English-only metadata for developers/model instructions.
- Friendly wording is scripted, not dynamically adapted to each patient.
- Realtime voice testing still depends on a valid `OPENAI_API_KEY`.

## Follow-Up Candidates

- Add tests that assert key clinical anchors remain present in rewritten prompt strings, such as `0 to 10`, urgent red-flag symptoms, and caregiver/proxy choices.
- Consider recording user feedback from live calls to tune wording further without changing the clinical state machine.
